# Развёртывание на VPS (AdminVPS)

Инструкция для Ubuntu / Debian. Для AlmaLinux/CentOS отличаются только команды
установки пакетов (`dnf` вместо `apt`) — остальное идентично.

Итоговая схема: **Nginx** принимает 80/443 → проксирует на **Next.js** (127.0.0.1:3000),
процесс держит **PM2**, сертификат выпускает **certbot**.

---

## 1. Подключиться к серверу

```bash
ssh root@IP_СЕРВЕРА
```

## 2. Базовая подготовка

```bash
apt update && apt upgrade -y
apt install -y curl git nginx ufw

# Файрвол: наружу только SSH и веб
ufw allow OpenSSH
ufw allow 'Nginx Full'
ufw --force enable
```

## 3. Node.js 22 LTS

```bash
curl -fsSL https://deb.nodesource.com/setup_22.x | bash -
apt install -y nodejs
node -v    # должно быть v22.x
npm -v
```

## 4. Отдельный пользователь для сайта

Гонять приложение от root небезопасно.

```bash
adduser --disabled-password --gecos "" momo
usermod -aG www-data momo
su - momo
```

## 5. Забрать код

Репозиторий приватный, поэтому нужен **deploy key** — ключ только на чтение.

```bash
# от пользователя momo
ssh-keygen -t ed25519 -C "momo-vps" -f ~/.ssh/id_ed25519 -N ""
cat ~/.ssh/id_ed25519.pub
```

Скопируйте вывод и добавьте на GitHub:
**репозиторий → Settings → Deploy keys → Add deploy key** (галочку «Allow write access» ставить НЕ нужно).

```bash
git clone git@github.com:e660eo/MOMO-Equipped.git ~/momo
cd ~/momo
```

## 6. Собрать и запустить

```bash
npm ci
npm run build

sudo npm install -g pm2      # если pm2 ещё не стоит
mkdir -p logs
pm2 start ecosystem.config.cjs
pm2 save
```

Автозапуск после перезагрузки сервера:

```bash
pm2 startup systemd -u momo --hp /home/momo
# выполните команду, которую напечатает pm2 (её нужно запустить от root)
```

Проверка, что приложение отвечает локально:

```bash
curl -I http://127.0.0.1:3000     # ожидаем HTTP/1.1 200 OK
```

## 7. Nginx

```bash
exit    # вернуться в root

cp /home/momo/momo/deploy/nginx.conf.example /etc/nginx/sites-available/momo
nano /etc/nginx/sites-available/momo     # заменить server_name на свой домен или IP

ln -s /etc/nginx/sites-available/momo /etc/nginx/sites-enabled/momo
rm -f /etc/nginx/sites-enabled/default
nginx -t && systemctl reload nginx
```

Теперь сайт открывается по IP сервера.

## 8. SSL (после переключения домена)

Сертификат выпускается только когда домен уже указывает на этот сервер.

```bash
apt install -y certbot python3-certbot-nginx
certbot --nginx -d momo-eq.ru -d www.momo-eq.ru
```

Certbot сам допишет конфиг Nginx и настроит автопродление.

## Обновление сайта

### Автоматическое (настроено)

Крон под пользователем `momo` раз в 3 минуты запускает
`scripts/auto-update.sh`: если в origin/main появились новые коммиты —
подтягивает, пересобирает и перезапускает без простоя. Когда новых коммитов
нет, скрипт выходит мгновенно.

Установка (один раз, под `momo`):

```bash
( crontab -l 2>/dev/null; echo '*/3 * * * * bash /home/momo/momo/scripts/auto-update.sh' ) | crontab -
crontab -l    # проверить, что строка появилась
```

Журнал обновлений: `~/momo/logs/auto-update.log`.
Отключить: `crontab -e` и удалить строку.

### Ручное (если нужно прямо сейчас)

```bash
su - momo
cd ~/momo
bash scripts/deploy.sh
```

Скрипт забирает код, ставит зависимости, пересобирает и перезапускает без простоя.

## Панель управления: папка данных и пароль

Панель (`/admin`) правит каталог прямо на сервере, поэтому данные лежат
**вне репозитория**: `deploy.sh` начинается с `git pull --ff-only`, и файлы
внутри `data/` каждое обновление затирались бы вместе с правками.

Настраивается один раз, под пользователем `momo`:

```bash
su - momo
mkdir -p ~/momo-data
```

Пропишите путь к ней и задайте пароль:

```bash
cd ~/momo
echo 'MOMO_DATA_DIR=/home/momo/momo-data' >> .env.local
node scripts/admin-password.mjs
```

Скрипт спросит пароль дважды (нажатия видны звёздочками) и сам допишет хеш
с секретом сессий в `.env.local`, выставив файлу права `600`. Переносить
длинные строки руками не нужно — при копировании в них легко потерять символ.
Сам пароль нигде не сохраняется: восстановить его нельзя, можно только задать
новый тем же скриптом. Нужны строки на экран (например, для другого сервера) —
запустите с флагом `--print`.

Логин — почта магазина из настроек сайта (`contacts.email`). Нужен отдельный
адрес, не совпадающий с публичным, — допишите в `.env.local` строку
`ADMIN_EMAIL=…`, она перекрывает почту из настроек.

Файл `.env.local` не в git — обновления сайта его не трогают. Осталось
перезапустить приложение, чтобы оно увидело переменные:

```bash
pm2 reload momo --update-env
```

После этого заходите на `https://momo-eq.ru/admin`.

Пока переменные не заданы, сайт работает на данных из репозитория как раньше,
а панель честно пишет, что не настроена, и ничего не сохраняет.

### Выйти из панели на всех устройствах

Вход в панель держится подписанной кукой на неделю, на сервере она нигде не
хранится — поэтому «разлогинить» её обычным способом нельзя. Если ноутбук
потерян, пароль скомпрометирован или просто хочется закрыть все открытые
входы, добавьте (или увеличьте на единицу) строку:

```bash
echo 'ADMIN_SESSION_VERSION=2' >> ~/momo/.env.local
pm2 reload momo --update-env
```

Все выданные до этого куки панели перестают работать мгновенно — включая
вашу, так что войти придётся заново. Значение — любое, важно лишь то, что
оно изменилось. По умолчанию `1`.

Куки покупателей это не трогает: у них своя. Кука покупателя умирает сама,
когда ему меняют пароль, — сброс пароля из раздела «Клиенты» заодно
выгоняет того, кто уже сидел в этом аккаунте.

### Что лежит в папке данных

```
~/momo-data/
  products.json, categories.json, brands.json, news.json, bundles.json, site.json
  orders.json — заказы с сайта, внутри персональные данные покупателей
  uploads/    — фото, загруженные через панель
  backups/    — последние 20 копий каждого файла, пишутся перед каждой правкой
```

⚠️ `orders.json` и копии в `backups/` содержат имена, телефоны и адреса
покупателей. Архивы с ними храните там, где к ним нет посторонних, и не
выкладывайте в общие облака.

Файлы появляются сами при первом обращении — копией из репозитория. Исходные
145 снимков каталога остаются в коде (`public/uploads`) и отдаются тем же
адресом `/media/…`, копировать их никуда не нужно.

## Письма о заказах

Каждый новый заказ уходит письмом владельцу: номер, состав, сумма, телефон
покупателя ссылкой (нажал — позвонил или открыл WhatsApp) и кнопка «Открыть
в панели». Пока ящик не подключён, сайт работает как раньше — заказ просто
виден только в разделе «Заказы».

**Письмо никогда не мешает заказу.** Оно отправляется уже после того, как
заказ сохранён, и в отдельном потоке: почтовый сервер молчит — покупатель
этого не замечает, а причина уезжает в лог и в панель.

### Шаг 1. Пароль приложения в Яндексе

Почта магазина на Яндексе, а он не пускает сторонние программы по обычному
паролю от ящика — нужен отдельный **пароль приложения**:

1. Зайдите на `id.yandex.ru` под ящиком магазина.
2. Раздел «Безопасность» → «Пароли приложений».
3. Создайте пароль для типа «Почта», назовите его, например, `Сайт momo-eq`.
4. Скопируйте показанные 16 символов — второй раз их не покажут.

Если в разделе пусто или пункт не находится, включите доступ по почтовым
программам в самой Яндекс.Почте: «Настройки» → «Почтовые программы» →
разрешить IMAP.

### Шаг 2. Прописать на сервере

Под пользователем `momo`, в том же `.env.local`, где уже лежат
`MOMO_DATA_DIR` и пароль панели:

```bash
cd ~/momo
cat >> .env.local <<'EOF'
SMTP_USER=mail@momo-eq.ru
SMTP_PASS=сюда16символовпароляприложения
EOF
pm2 reload momo --update-env
```

Всё остальное подставится само: сервер `smtp.yandex.ru`, порт `465`,
отправитель — тот же ящик, получатель — он же.

Нужны другие значения — допишите рядом:

| Переменная | По умолчанию | Зачем менять |
| --- | --- | --- |
| `ORDER_NOTIFY_TO` | адрес из `SMTP_USER` | слать заказы на другой ящик; несколько — через запятую |
| `SMTP_HOST` | `smtp.yandex.ru` | почта не на Яндексе |
| `SMTP_PORT` | `465` | `587`, если провайдер требует STARTTLS |
| `SMTP_FROM` | `MOMO — сайт <SMTP_USER>` | другая подпись отправителя |

⚠️ Знак `$` в пароле приложения нужно писать как `\$`: значения из `.env`
проходят подстановку переменных, и всё после `$` съелось бы. У Яндекса
пароли из одних латинских букв, так что обычно это не встречается.

### Шаг 3. Проверить

Панель → «Контакты и условия» → блок **«Письма о заказах»** →
«Отправить пробное письмо». Панель напишет, ушло ли оно, а при отказе —
что именно проверить.

Если письмо о настоящем заказе не уйдёт, на главной странице панели появится
красное предупреждение: заказы приходят, а почта молчит — узнать об этом
нужно сразу, а не через неделю.

## Оплата на сайте (Яндекс Пэй и Сплит)

Покупатель платит картой или Сплитом сразу в корзине. Заказ сохраняется до
оплаты, поэтому брошенная на полпути оплата не теряет заявку: она просто
висит неоплаченной.

⚠️ **Оплачивается только товар.** Стоимость доставки на момент оформления
неизвестна, поэтому в платёж она не входит — менеджер согласует её отдельно.
Это же написано покупателю в корзине и в карточке заказа в панели.

### Чего требует закон

**Без онлайн-кассы принимать оплату нельзя.** Яндекс Пэй — платёжный сервис,
фискальные чеки он не выдаёт; по 54-ФЗ чек обязан пробить продавец. Штраф —
от четверти до половины суммы расчёта (ст. 14.5 КоАП).

Покупать кассовый аппарат не нужно, подойдёт облачная: регистрация в АТОЛ
Онлайн или CloudKassir, затем в кабинете Яндекс Пэй **Настройки → Облачная
касса → Подключить облачную кассу**. После этого чеки уходят покупателю и в
налоговую сами.

### Что настроить в кабинете Яндекс Пэй

Кабинет — `console.pay.yandex.ru`. Вверху страницы обязательно выберите
магазин **momo-eq.ru**: в аккаунте есть второй, на мёртвом домене mo.net.ru.

1. **Сплит** — меню «Сервисы», выбрать Сплит, подать заявку, заключить
   договор. Без него на форме будет только карта; код при этом менять не
   надо, Яндекс сам покажет доступное.
2. **Callback URL** — в «Настройках» вписать адрес, куда приходят уведомления
   об оплате:

   ```
   https://momo-eq.ru/api/pay/callback
   ```

   Тот же адрес показан в панели сайта, в блоке «Оплата на сайте».
3. **Ключ Merchant API** — оттуда же. В тестовом режиме он совпадает с
   Merchant ID, в боевом это отдельная строка.

⚠️ **Боевой ключ не пересылать никому, включая меня.** Он равнозначен праву
принимать деньги от имени магазина. Его место — только файл `.env.local` на
сервере. Тестовый ключ безопасен: им ничего не оплатить.

### Что прописать на сервере

Под пользователем `momo`, в тот же `.env.local`:

```bash
cd ~/momo
cat >> .env.local <<'EOF'
YANDEX_PAY_MERCHANT_ID=идентификатор-магазина
YANDEX_PAY_API_KEY=ключ-merchant-api
EOF
pm2 reload momo --update-env
```

Этого достаточно для **тестового режима**: в корзине появится кнопка
«Оплатить на сайте», но платежи будут ненастоящими, о чём честно написано и
покупателю, и в панели.

Тестовые суммы Яндекса: **10 000 ₽** — успешная оплата, **10 001 ₽** — сбой,
**10 002 ₽** — не хватило денег.

### Переход на настоящие деньги

Только после того, как заключён договор на Сплит и подключена онлайн-касса:

```bash
cd ~/momo
echo 'YANDEX_PAY_LIVE=1' >> .env.local
# и заменить YANDEX_PAY_API_KEY на боевой
pm2 reload momo --update-env
```

Проверить, что режим сменился, — в панели: блок «Оплата на сайте» перестанет
показывать предупреждение о тестовом режиме.

**Пока переменной нет, сайт работает в песочнице.** Это намеренно: случайно
принять настоящие деньги на недонастроенном магазине хуже, чем случайно не
принять.

### Как это устроено внутри

- `src/lib/yandex-pay.ts` — создание заказа, запрос статуса, проверка подписи.
- `src/app/api/pay/callback/route.ts` — приём уведомлений.
- Уведомление проверяется дважды: подпись ES256 ключом Яндекса, затем
  перезапрос статуса по API. Письму «оплачено» на слово не верим — иначе
  заказы помечал бы оплаченными любой, кто знает адрес вебхука.
- Страница возврата `/order/<номер>` открывается только по случайной метке из
  ссылки: номера заказов идут подряд и перебираются за минуту, а на странице
  видны данные покупателя.
- Об оплате приходит отдельное письмо — заказ и деньги приходят не
  одновременно.

### Резервная копия

```bash
tar czf ~/momo-data-$(date +%F).tar.gz -C /home/momo momo-data
```

Восстановление — распаковать архив обратно в `/home/momo` и перезапустить:
`pm2 reload momo`. Откатить одну неудачную правку можно из `backups/`:
скопировать нужную копию поверх файла и сделать `pm2 reload momo`.
